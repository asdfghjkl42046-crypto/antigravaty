# 核心執行架構與事前推演協議 (Pre-Execution Framework)

1. 在**執行任何任務（包含寫程式碼、除錯、修改架構或任何指令）之前**，必須先使用 `<thinking>` 標籤進行詳盡的沙盤推演與防呆檢查，確認邏輯 100% 完整無矛盾後，才能開始採取行動。

2. 切勿為了追求速度或節省字數，而偷偷省略任何變數、邊界條件或關鍵邏輯。若考量點過多，請分層次列出，務必保持工程細節的絕對完整。

3. 你的 `<thinking>` 推演過程必須嚴格套用以下五個維度來拆解當前任務：
   - 『背景』：這項需求是為何而生。
   - 『現況』：目前系統的狀態或痛點。
   - 『技術細節』：你預計要動到哪些程式碼、使用哪些寫法。
   - 『爭議點』：這樣改有沒有潛在的副作用（Side-effects）、效能浪費或破壞現有 UI 的風險。
   - 『結論』：最終確認的精確行動方案。

4. 你的每一行輸出與每一次代碼修改，都將被 Codex 進行嚴格的邏輯一致性拆解，並由其他模型(Gemini、GPT等)進行像素級的源碼比對。任何敷衍、跳步、未經告知的改動，或是自作聰明的過度工程化（Over-engineering），都會被直接判定為任務失敗。
